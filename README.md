# FF4 Battle Auto Bot

## 概要
Final Fantasy IV の戦闘を対象に、画面上のUIを画像認識しながら操作を行う自動化Botです。  
移動によるエンカウント発生から、戦闘開始判定、通常攻撃、HP低下時の回復、勝利後の進行までを自動化しています。

本Botは、単純なキー送信ではなく、ゲーム画面の状態に応じて処理を切り替えることで、ゲーム特有の状態変化を伴う操作自動化に取り組んだ個人開発です。

## 背景・目的
業務でE2Eテスト自動化に携わる中で、繰り返し実行できる自動化の価値を感じる一方、UI変更や仕様変更によってテストコードの保守が継続的に発生する難しさを経験してきました。

その延長として、より状態変化の大きい対象であるゲームに着目し、画面認識を用いた戦闘Botを個人開発しました。  
特にゲームでは、同じ処理を繰り返すだけでは成立せず、画面状態に応じて振る舞いを変える必要があります。そこで本Botでは、戦闘開始・戦闘中・回復・戦闘終了といった状態ごとに処理を分ける構成を取りました。

## 実装内容
- 左右移動による敵エンカウント
- セシルUIおよびATBフレームの検出による戦闘開始判定
- 攻撃コマンドの検出後、自動で通常攻撃を実行
- セシルのHP低下を検知した場合、ポーションを使用して回復
- 勝利画面を検出後、進行用の入力を自動実行
- ESCキーによる強制停止

## 設計上の工夫
- 画像認識失敗で処理全体が停止しないよう、例外を吸収する関数を用意
- 戦闘開始判定では、戦闘UIらしき要素を検出したあとに、攻撃ボタンが表示されるまで複数回確認することで、画面遷移直後の不安定さを考慮
- 回復処理では、アイテムや対象キャラクターが見つからない場合にメニューを戻して処理を終了するようにし、誤操作の連鎖を防止
- 一連の処理を「移動中」「戦闘開始待ち」「戦闘中」「回復」「勝利後」に分け、状態ごとに振る舞いを切り替える構造とした

## この開発で直面した課題
本Botは通常戦闘の自動化には成功した一方で、画像認識と条件分岐を中心としたルールベースの自動化には限界があることも明確になりました。

例えば、以下のような状況では動作が不安定になりやすいと考えています。

- バックアタックなど通常と異なるUI配置
- キャラクター位置や表示順の変化
- 一時的な演出や画面遷移による誤検出
- 想定していない戦闘状態やメニュー遷移

この経験から、ゲームにおける自動化では単純な手順再現だけでなく、現在の状態をより柔軟に解釈し、状況に応じて振る舞いを変える仕組みが重要だと考えるようになりました。

## 今後の改善
- バックアタックなど通常戦闘と異なるレイアウトを異常状態として検知し、自動停止する機能
- 異常発生時のスクリーンショット保存
- ログ出力の整理と失敗原因の追跡性向上
- 複数キャラクターや異なる戦闘状況への対応
- Jenkins等を用いた定期実行への拡張
- 将来的には、画面や状態をより柔軟に解釈できる仕組みを取り入れ、変化に強い自動化へ発展させることを検討
